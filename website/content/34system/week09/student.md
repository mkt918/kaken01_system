## 🕒 レッスン 1: データベースとは？Excelとの違い

### 📊 「見た目」は似ているが、「中身」は全く違う

データベースは、ExcelやGoogleスプレッドシートによく似た「表（テーブル）」の形をしています。
しかし以下の点で大きく異なります。

| 項目 | Excel | データベース |
|------|-------|-----------|
| **使う人数** | 1人（基本）が1つの画面 | **何千人・何万人が同時にアクセス** |
| **速度** | 遅い（ファイルサイズが大きいと重くなる） | **極めて高速**（何百万件のデータでもサクサク） |
| **データ量** | 最大100万行程度が限界 | **数兆件のデータも管理可能** |
| **信頼性** | ファイルが壊れるリスクがある | **堅牢設計・自動バックアップ** |
| **複数アクセス時** | 同時編集するとデータが壊れる危険 | **同時アクセスで自動に調整** |

#### 🎮 実例：オンラインゲーム「ドラゴンクエスト」

```
【もしExcelで作ったら】
プレイヤーA: 攻撃を使う → プレイヤーBのHP更新
          ↓
プレイヤーB: 同時に魔法を使う → プレイヤーAのHP更新
          ↓
エラー！「ファイルが競合しています」→ データ消失😭

【データベースなら】
プレイヤーA: 攻撃を使う → 自動的に順序付けられる
          ↓
プレイヤーB: 同時に魔法を使う → 自動的に順序付けられる
          ↓
完璧に処理される。全員のデータ安全✨
```

### 📊 データベースの基本用語（絶対に覚えよう！）

データベースには、覚えなければいけない「魔法の言葉」があります。これらは世界中すべてのプログラマーが使う用語です。

#### 【用語1】テーブル (Table)

**テーブル** = 「データのまとまり」＝ 1つの「表」全体

```
【heroes テーブル】
id | name         | hp
---|--------------|----
1  | アーサー     | 100
2  | ランスロット | 80
3  | ガウェイン   | 75
```

- Excelの「1つのシート」に相当する
- 1つのテーブルには「同じカテゴリー」のデータだけが入る
- ゲームなら：`heroes`テーブル、`monsters`テーブル、`items`テーブル...など複数存在
- 銀行なら：`customers`テーブル、`accounts`テーブル、`transactions`テーブル...など

#### 【用語2】カラム (Column/列)

**カラム** = 「縦の項目」＝ 1つの「属性」

```
           ↓ これがカラム（列）
【heroes テーブル】
id | name         | hp    ← ここまで全部が「カラム」
---|--------------|----
1  | アーサー     | 100
2  | ランスロット | 80
```

- 「何のデータか」を示す：名前、年齢、点数、HPなど
- Excelの「列の見出し」（A列、B列）に相当
- 各カラムには「データ型」（数字か文字か）が決められている
  - `id`: INTEGER（整数）
  - `name`: TEXT（文字列）
  - `hp`: INTEGER（整数）

#### 【用語3】レコード (Record/行)

**レコード** = 「横のデータ1件分」＝ 1つの「個体」のすべての情報

```
                          ↓ これが1つのレコード（行）
【heroes テーブル】
id | name         | hp
---|--------------|----
1  | アーサー     | 100  ← ここ全体（1人分）
2  | ランスロット | 80   ← ここ全体（1人分）
3  | ガウェイン   | 75   ← ここ全体（1人分）
```

- 1つの「個体」「トランザクション」「イベント」のすべての情報を表す
- Excelの「1行分」に相当
- 例：
  - ゲーム：1人のプレイヤーの全データ（Lv、HP、MP、経験値...）
  - 銀行：1件の取引（日時、金額、種類、口座番号...）
  - 学校：1人の生徒の全データ（名前、学籍番号、クラス、成績...）

#### 🔗 3つの関係をシンプルに理解

```
【students テーブル】
id | name     | grade      ← これら全部が「カラム」
---|----------|------
1  | たろう   | 1          ← これが「レコード」（1人分）
2  | はなこ   | 2          ← これが「レコード」（1人分）
3  | さぶろう | 1          ← これが「レコード」（1人分）
```

> **💡 覚え方のコツ**
> 
> - **テーブル**：「表」全体（1つの表データ）
> - **カラム**：「縦」（Column = Column = 縦方向）
> - **レコード**：「横」「1件」（Record = 記録・1つの記録）
> 
> 「クラスの名簿Excel」で例えると：
> - テーブル = シート全体
> - カラム = 「出席番号」「名前」「生年月日」...（列タイトル）
> - レコード = 「太郎君のデータ」「花子さんのデータ」...（1行分）

## 📝 補足：データ型について（重要！）

SQLでテーブルを作るとき、**「どんなデータが入るのか」を先に決める**必要があります。これを「型」と言います。

### 🎯 主なデータ型

| 型 | 名前 | 例 | 説明 |
|------|------|-----|------|
| `INTEGER` | 整数型 | `10`, `-5`, `0` | 小数点なしの数字のみ |
| `TEXT` | 文字型 | `"太郎"`, `"200円"` | 文字列。何文字でもOK |
| `REAL` | 実数型 | `3.14`, `165.5` | 小数を含む数字 |
| `BOOLEAN` | 真偽値 | `TRUE`, `FALSE` | 真（○）か偽（×）か |

### 🤔 「型を決める」ってなぜ必要？

#### Pythonとの違い

```python
# 【Pythonの場合】簡単だが自由すぎる
age = 17           # 整数でOK
age = "17"         # 文字列でもOK（型が自動判定）
age = 17.5         # 小数でもOK（型が自動判定）

# ↓ 結果：計算時にトラブル
print(age + 1)     # "17" なら、エラー！
```

```sql
-- 【SQLの場合】厳格だが堅牢
CREATE TABLE students (
    age INTEGER    -- 「この列は整数だけ」と宣言
);

INSERT INTO students (age) VALUES (17);      -- OK ✓
INSERT INTO students (age) VALUES ('17');    -- エラー！('は文字に見えるから)
INSERT INTO students (age) VALUES (17.5);    -- エラー！(小数は INTEGER に入らない)
```

### 📌 型を決めるメリット

1. **データを守る**：間違ったデータが混入しない
   - 「年齢」は数字だけ→ 間違い防止
   - 「名前」は文字だけ→ 名前に数字が入ることはない

2. **計算が確実**：型が正しければ計算で失敗しない
   - INTEGER 同士の足し算 = INTEGER（確実）
   - 小数を含む計算が必要なら REAL を使う（確実）

3. **データを効率的に保存**：メモリを無駄にしない
   - INTEGER：4バイト（決まった大きさ）
   - TEXT：可変長（必要な分だけ使う）

### 💡 型の選び方チェックリスト

```
「年齢」「個数」「ID番号」「LV」「HP」 → INTEGER（整数）
「身長」「体重」「気温」「金額（円未満）」 → REAL（小数）
「名前」「住所」「商品名」「説明文」 → TEXT（文字）
「ログイン済み」「休場予定」「合格」 → BOOLEAN（◯/×）
```

## 🕒 レッスン 2: 最初のテーブルを作ってみよう（CREATE TABLE）

### 📚 SQLの最初の「魔法の呪文」

テーブルを作るときにPythonでは単に `name = "たろう"` と書きましたが、SQLでは「どんな名前の列（カラム）を、どんなデータ型で作るか」をキッチリ宣言する必要があります。

この宣言を「**スキーマ設計**」と言います。

### 💻 ハンズオン（オンラインSQLで実行しよう！）

先生が指定した「SQLを練習できるウェブサイト（sqliteonline.comなど）」を開き、左側の入力欄に以下の魔法の呪文（SQL）を書いて実行（Run）してください。

#### 【ステップ1】テーブルの設計図を書く

```sql
/* 生徒（students）のデータを保存するテーブルを作る */
CREATE TABLE students (
    id INTEGER,      /* 数字を入れる列 */
    name TEXT,       /* 文字を入れる列 */
    grade INTEGER    /* 学年（数字）を入れる列 */
);
```

#### 📖 CREATE TABLE 文の詳しい構文

```
CREATE TABLE テーブル名 (
    カラム名1 データ型1,
    カラム名2 データ型2,
    カラム名3 データ型3
);
```

**各部分の意味：**

| 部分 | 意味 |
|------|------|
| `CREATE TABLE` | 「新しいテーブルを作る」という命令（魔法の最初の言葉）|
| `students` | テーブル名（小文字で、単数形よりも複数形がSQLの慣例） |
| `id INTEGER` | 「id という列を作る、中身は整数」 |
| `name TEXT` | 「name という列を作る、中身は文字」 |
| `,`（カンマ） | 列と列の区切り。**最後の列の後には付けない** |
| `;`（セミコロン） | 「このSQL文はここで終わり」の合図。**絶対に忘れない** |

> **💡 解説ポイント**
>
> - `CREATE TABLE` は「テーブルを作れ！」という命令です。
> - `INTEGER`（インティジャー）は「整数・数字」、`TEXT`（テキスト）は「文字」という意味です。
> - データベースは型（文字か数字か）にとても厳しいです。「id に文字を入れろ」とやると**エラー**になります。
> - 一番おしりには必ず `;`（セミコロン）をつけます。これを忘れるとエラーになります。

#### ⚠️ よくあるミス（BEST 3）

```sql
❌ ミス1：最後の）の後にセミコロンがない
CREATE TABLE students (
    id INTEGER,
    name TEXT,
    grade INTEGER
);  ← セミコロン必須！

❌ ミス2：最後のカラムの後にカンマがある
CREATE TABLE students (
    id INTEGER,
    name TEXT,
    grade INTEGER,  ← このカンマは要らない
);

❌ ミス3：データ型を忘れる
CREATE TABLE students (
    id,         ← INTEGER がないとエラー！
    name TEXT,
    grade INTEGER
);
```

**覚え方：「最後のカンマはなし、セミコロンはあり」**

## 🕒 レッスン 3: 倉庫にデータを投げ込む（INSERT）

### 📚 テーブルができたら、次はデータを入れる

テーブルという「枠」ができたので、実際にデータ（レコード）を入れていきましょう！
データを入れる魔法は `INSERT INTO`（インサート・イントゥ）です。

### 💻 ハンズオン（やってみよう）

以下のSQLを実行して、さきほど作った `students` テーブルに3人のデータを入れてみましょう。

```sql
/* テーブルにデータを追加する（1人目） */
INSERT INTO students (id, name, grade) VALUES (1, 'たろう', 1);

/* テーブルにデータを追加する（2人目） */
INSERT INTO students (id, name, grade) VALUES (2, 'はなこ', 2);

/* テーブルにデータを追加する（3人目） */
INSERT INTO students (id, name, grade) VALUES (3, 'さぶろう', 1);

/* 【データ確認用】テーブルの中身をすべて見る */
SELECT * FROM students;
```

#### 📖 INSERT文の詳しい構文

```
INSERT INTO テーブル名 (カラム名1, カラム名2, カラム名3)
VALUES (値1, 値2, 値3);
```

**各部分の意味：**

| 部分 | 意味 |
|------|------|
| `INSERT INTO` | 「データを追加する」という命令 |
| `students` | テーブル名（どのテーブルに入れるか） |
| `(id, name, grade)` | どのカラムに入れるか（順番が重要！） |
| `VALUES` | 「ここからが値だよ」という区切り言葉 |
| `(1, 'たろう', 1)` | 実際に入れる値。**個数と順番が一致していること** |

#### 📌 重要ルール：「型」と「引用符」

```sql
-- 【INTEGER型のカラム】
INSERT INTO students (id) VALUES (1);      -- OK ✓ 数字をそのまま
INSERT INTO students (id) VALUES ('1');    -- エラー！ 引用符で囲んではダメ

-- 【TEXT型のカラム】
INSERT INTO students (name) VALUES ('太郎');    -- OK ✓ シングル引用符で囲む
INSERT INTO students (name) VALUES ("太郎");    -- エラー！ ダブル引用符はNG
INSERT INTO students (name) VALUES (太郎);      -- エラー！ 引用符なしはNG
```

**引用符の規則：**

| 型 | 例 | ルール |
|------|------|------|
| INTEGER（数字） | `1`, `100`, `-5` | **引用符なし**。数字だけ |
| TEXT（文字） | `'太郎'`, `'りんご'` | **シングル引用符 `'` で囲む**。`"` はダメ |

> **💡 Python との違い**
> 
> Pythonでは `"太郎"` も `'太郎'` も同じでしたが、**SQLではシングル引用符 `'` のみ**が文字列を表します。これは大事なルールです！

#### ⚠️ よくあるミス（BEST 3）

```sql
❌ ミス1：TEXT のシングル引用符を忘れる
INSERT INTO students (id, name, grade) VALUES (1, たろう, 1);
                                                  ↑ 引用符がない → エラー！

❌ ミス2：ダブル引用符を使う（SQLはシングルのみ）
INSERT INTO students (id, name, grade) VALUES (1, "たろう", 1);
                                                  ↑↑ ダブルはダメ

❌ ミス3：セミコロンを忘れる
INSERT INTO students (id, name, grade) VALUES (1, 'たろう', 1)
                                              ↑ セミコロン必須！
```

**覚え方：「TEXT は ' で囲む、文の終わりに ; 」**

#### 📚 おまけ：「データ確認」の魔法 SELECT

さきほど使った以下の命令：

```sql
SELECT * FROM students;
```

これは「students テーブルのすべてのデータを表示する」という意味です。

| 命令部分 | 意味 |
|---------|------|
| `SELECT` | 「データを選んで表示する」という命令 |
| `*` | 「すべての列」という意味の記号 |
| `FROM` | 「どのテーブルから取るか」という区切り言葉 |
| `students` | テーブル名 |

**結果は表形式で表示されます：**

```
id | name     | grade
---|----------|------
1  | たろう   | 1
2  | はなこ   | 2
3  | さぶろう | 1
```

## 🚀 早く終わった人へ：ミニプロジェクト「ファンタジーRPG・モンスター図鑑」

キミはゲーム開発者です！今回はプレイヤーが戦う前に確認できる「モンスター図鑑」のデータベースを構築します。

### 📝 ミッション（モンスター図鑑）
