# 指導案: Week 24 「全員参加のバグ探し大会（テスト）」

## 🎯 今週のねらいと授業のゴール
開発が佳境に入ると、生徒は「自分だけのテストデータ」でしか動作確認をしなくなり、本番の発表でパニックを起こします。
他人に触らせる（QAテスト：Quality Assurance）経験を通じて、**「プログラミングにおける『想定外の使われ方』の恐ろしさ」**を実感させると同時に、それを修正する「堅牢なシステム作り」の重要性を教えます。

---

## ⏱️ タイムテーブル (180分)

### 1️⃣ テストの意義とルールの説明 (0分〜15分)
- **【解説】**: 「マリオが壁にめり込むバグを探すお仕事（デバッグ）があるのを知ってる？」とゲーム開発を例に出して、QAテストがプロの立派な職業であることを伝えます。
- **【ルール設定】**: 「今日は、友達のアプリがエラーを吐いて停止したら、見つけた人の『勝ち』です！いじわるな客になりきってください！」と宣言し、ゲーム感覚を持たせます。

### 2️⃣ 席替え・相互破壊テスト開始 (15分〜60分)
- **【演習】**: 「右隣の席のPCに移動！」と号令をかけ、テストを実施させます。
- **【指導のポイント】**: 
  - 絶対に「ソースコード（Pythonの文字）」を直接いじってはいけません。あくまで「マウスとキーボードの入力（黒い画面やGUI）」だけでフリーズさせるように徹底させます。
  - いじわるな入力（文字数制限オーバー、空欄、変な記号）を生徒同士で競わせて、教室内を大いに盛り上げてください。

### 3️⃣ バグリポートの提出 (60分〜70分) ※その後休憩10分
- **【演習】**: バグを見つけた経緯と、アプリ自体の感想（褒め言葉）をセットにして、付箋紙やノートの切れ端に書いて元の席の持ち主に渡させます。
- **【配慮事項】**: 「俺のアプリ壊しやがって…」と喧嘩になってはいけません。「本番前に弱点を教えてくれた恩人だ！」というマインドセットを必ず醸成してください。

### 4️⃣ ひたすらデバッグ（修正） (80分〜180分)
- **【演習】**: もらったバグリポートをもとに、修正プログラムを書かせます。
- **【教員からの鉄板の助け舟】**:
  空欄エラーで落ちる生徒が続出します。以下の `if` 文のガードの仕方を黒板に書いてあげてください。
  ```python
  text = entry.get()
  
  # ▼ 何も入力されていない（空文字の）場合は、returnして処理を強制終了するガード！
  if text == "":
      # tk.messagebox.showerror("エラー", "名前を入力してね！") # （表示できれば）
      return 
  ```
  この1行を知るだけで、システムの安定性が100倍上がります。
